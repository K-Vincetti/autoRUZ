# ДЗ: MVCC, системные поля строк, блокировки и VACUUM

## Начальное состояние строки

Команда:

SELECT xmin, xmax, ctid, * FROM mvcc_test;

![[Screenshot 2026-03-11 112819.png]]

Результат:

|xmin|xmax|ctid|id|value|
|---|---|---|---|---|
|785|0|`(0,1)`|1|`v1`|

Вывод:

Строка актуальна, так как `xmax = 0`.

---

## Обновление строки

Команды:

UPDATE mvcc_test  
SET value = 'v2'  
WHERE id = 1;

SELECT xmin, xmax, ctid, * FROM mvcc_test;

![[Screenshot 2026-03-11 113300.png]]

Результат:

|xmin|xmax|ctid|id|value|
|---|---|---|---|---|
|787|0|`(0,2)`|1|`v2`|

Вывод:

После обновления `xmin` изменился с `785` на `787`, `ctid` изменился с `(0,1)` на `(0,2)`.  
Появилась новая версия строки.

---

## Просмотр страницы через pageinspect

Команда:

SELECT lp, t_xmin, t_xmax, t_ctid, t_infomask, t_infomask2  
FROM heap_page_items(get_raw_page('mvcc_test', 0));

![[Screenshot 2026-03-11 113347.png]]

Результат:

|lp|t_xmin|t_xmax|t_ctid|t_infomask|t_infomask2|
|---|---|---|---|---|---|
|1|785|787|`(0,2)`|1282|16386|
|2|787|0|`(0,2)`|10498|32770|

Вывод:

На странице видно две версии строки.

Старая версия: `t_xmin = 785`, `t_xmax = 787`.  
Новая версия: `t_xmin = 787`, `t_xmax = 0`.

`t_infomask` отличается у старой и новой версии: `1282` и `10498`.

---

## Что хранится в t_infomask

`t_infomask` хранит служебные флаги состояния строки.

По результатам видно:

|Версия|t_infomask|Состояние|
|---|---|---|
|Старая|1282|заменена|
|Новая|10498|актуальна|

Вывод:

`t_infomask` показывает внутреннее состояние версии строки: актуальна она, заменена, удалена или заблокирована.

## Разные транзакции

Команды в первой транзакции:

BEGIN;

UPDATE mvcc_test  
SET value = 'v3'  
WHERE id = 1;

SELECT xmin, xmax, ctid, * FROM mvcc_test;

![[Screenshot 2026-03-11 113529.png]]

Результат в первой транзакции:

|xmin|xmax|ctid|id|value|
|---|---|---|---|---|
|788|0|`(0,3)`|1|`v3`|

Результат во второй транзакции:

|xmin|xmax|ctid|id|value|
|---|---|---|---|---|
|787|788|`(0,2)`|1|`v2`|

Вывод:

Первая транзакция видит новую версию строки `v3`.

Вторая транзакция видит предыдущую версию `v2`.

---

## Конфликт транзакций

В первой транзакции:

BEGIN;

UPDATE mvcc_test  
SET value = 'a1'  
WHERE id = 1;

UPDATE mvcc_test  
SET value = 'a2'  
WHERE id = 2;

![[Screenshot 2026-03-11 115225.png]]

Во второй транзакции:

UPDATE mvcc_test  
SET value = 'zzz'  
WHERE id = 1;

![[Screenshot 2026-03-11 115329.png]]

Результат:

ERROR: current transaction is aborted, commands ignored until end of transaction block

Вывод:

При конфликтующих обновлениях транзакция перешла в состояние ошибки.  
После этого нужно выполнить `ROLLBACK`.

## Очистка данных

Команды:

UPDATE mvcc_test SET value = 'x1';  
UPDATE mvcc_test SET value = 'x2';  
UPDATE mvcc_test SET value = 'x3';

VACUUM mvcc_test;

![[Screenshot 2026-03-11 115643.png]]

Результат:

VACUUM

Вывод:

После нескольких обновлений была выполнена очистка.  
`VACUUM` завершился успешно.

---

## Краткий итог

После `UPDATE` создаётся новая версия строки.

`xmin` показывает транзакцию, которая создала версию строки.

`xmax` показывает транзакцию, которая заменила или удалила версию строки.

`ctid` показывает физическое положение версии строки.

`t_infomask` хранит внутренние флаги состояния строки.

В разных транзакциях могут быть видны разные версии одной строки.

При конфликтующих обновлениях возникает ошибка транзакции.

`VACUUM` очищает старые версии строк.
